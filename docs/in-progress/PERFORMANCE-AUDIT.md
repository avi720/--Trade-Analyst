## Background

Trade Analyst is a Hebrew-RTL trading journal for self-directed equity traders, built on Next.js 16 (App Router) + React 19 + Supabase Postgres with RLS. It pulls trade executions from IBKR's Flex Web Service, normalises them through a FIFO matching pipeline, and serves a research dashboard plus an AI chat assistant ("◊ó◊Ý◊ü", Gemini-backed). The app is deployed single-user today but architected multi-user at the DB layer per the invariant in `CLAUDE.md`, so all performance findings are framed against a power-user dataset (thousands of trades, hundreds of executions per IBKR sync) ‚Äî not a single-trade demo.

**Scope:** Runtime performance of the live request paths ‚Äî IBKR ingest, FIFO matching, research-dashboard render loop, trade-search render loop, AI chat context plumbing, and the cron handlers. Out of scope: correctness, security, and dependency hygiene (covered in [`docs/TECH-DEBT.md`](TECH-DEBT.md)); UI polish (covered in [`docs/UI-AUDIT.md`](UI-AUDIT.md)); cold-start / bundle-size analysis (not a current pain point on Vercel).

**Stack reviewed:** Next.js 16 (App Router) ¬∑ React 19 ¬∑ TypeScript ¬∑ Supabase JS 2.x ¬∑ `@supabase/ssr` ¬∑ `@google/genai` (Gemini 2.5 Flash + Pro) ¬∑ `recharts` ¬∑ `exceljs` ¬∑ `fast-xml-parser` ¬∑ Vercel hosting ¬∑ GitHub Actions cron.

**Methodology:**

## Strengths ‚Äî What Already Works Well

Preserve these patterns when refactoring:

- The partial unique index `Trade_userId_ticker_open_unique` on `("userId", ticker) WHERE status='Open'` (`CLAUDE.md` "FIFO logic ‚Äî invariants") makes the optimistic-concurrency retry in `process-executions.ts` cheap ‚Äî no pessimistic lock, no global mutex, conflicts self-heal in one re-read.
- The research dashboard already memoises the expensive derivations: `filteredTrades`, `stats`, and `chartData` are all wrapped in `useMemo` with correct dependency lists (`components/research-dashboard.tsx:143`, `:146`, `:182`). The bottleneck is downstream of these memos, not in them.
- `processExecutions` performs the `brokerExecId` dedup against the per-user UNIQUE constraint with a single `maybeSingle()` round-trip per execution (`lib/ibkr/process-executions.ts:171-176`) ‚Äî the right primitive; it just needs to fan out across tickers.
- `recomputeActualR` operates per-`tradeId` rather than scanning the whole user's `Trade` table, so the manual-import recompute pass scales with affected trades, not total trades.
- IBKR backfill uses `waitUntil()` from `@vercel/functions` (`CLAUDE.md` "Backfill / cron behavior") so the HTTP response returns 202 immediately and the long work continues asynchronously ‚Äî exactly the right shape for a multi-minute job under Vercel's request envelope.
- The IBKR cron poll is bounded (4 attempts √ó 10s) and degrades gracefully via `IbkrTransientError` rather than hot-looping, so a slow Flex statement doesn't burn the function's 60s budget repeatedly.
- The Activity Flex query is end-of-day, so cron runs only 2√ó/day at 13:00 & 20:00 UTC (`CLAUDE.md` "Single Flex Query"). No over-polling ‚Äî the sequential-by-user issue in P5 is the only thing standing in the way of this being a clean design.
- `parse-flex-xml.ts` resolves the PascalCase / camelCase Flex quirk in a single helper (`resolveStatement` + per-field `??` fallback) ‚Äî keeps the parse-time cost flat at O(executions) with no double-walk.

### Phase 1 ‚Äî Critical (Day-1 blockers, must clear before next release)

#### [ ] P1. Chat AI context ‚Äî full re-spec across both modes (was: "full mode sends unbounded JSON dump")
- **Where:** `app/api/chat/route.ts`, `components/chat-sidebar.tsx` (mode selector + new Pro-only filter-respect toggle), `components/research-dashboard.tsx` (contextData construction), `lib/chat/gemini-client.ts`, `lib/utils/research-aggregate.ts` (reused/extended for tool-use aggregation), new tool-definition modules for `queryTrades` + the aggregation tools (TBD at implementation time).
- **Issue:** **Reopened 2026-07-19** ‚Äî was Deferred 2026-06-19 pending a product decision; owner reversed and worked through a full characterization with the assistant, covering both chat modes. Original finding: `full` mode ran an unbounded `supabase.from('Trade').select(...)` with no `.limit()`, then `JSON.stringify()`'d the result straight into the `gemini-2.5-pro` system prompt on every message ‚Äî for a power user with 2-3k closed trades, a ~400-600 KB blob per turn. Follow-up investigation during characterization surfaced that `smart` mode has the **same unbounded-growth shape** when no dashboard filter is active (`filteredTrades === closedTrades` = full history) ‚Äî just with a slimmer per-trade field set and the cheaper `gemini-2.5-flash` model. The original finding undersold smart mode as "safe by construction"; it isn't.
- **Resolved specification ‚Äî Smart mode (`gemini-2.5-flash`):**
  - **Fields sent inline per trade:** `ticker, direction, actualR, result, setup, realizedPnl, closedAt`. `openedAt` intentionally excluded ‚Äî reserves hold-time-vs-R and day/hour-by-open-time analysis for Pro/Full. **Depends on P12** (day/hour charts must bucket by `openedAt`, not `closedAt`, for this gate to actually hold ‚Äî today Flash already receives `closedAt` and could reconstruct the *current* closedAt-based day/hour breakdown itself).
  - **Chart coverage:** all research charts answerable except hold-time-vs-R and day/hour (both blocked by the missing `openedAt` field, contingent on P12).
  - **Fallback threshold:** 60 KB (~17K tokens) on `JSON.stringify(trades)`. Starting estimate, not measured ‚Äî validate empirically (soak test below) and retune if needed.
  - **Above threshold, Pro subscribers:** switch to the shared tool-use architecture (below), exposing only Smart's field set ‚Äî no `openedAt`, `executionQuality`, `emotionalState`, or free-text fields.
  - **Above threshold, non-Pro:** shown a message instead of an automatic fallback (exact copy TBD at implementation). Gated to Pro because of the extra Gemini round-trips (added latency/complexity), **not** token cost ‚Äî tool-use is actually cheaper in total tokens than the raw dump for large datasets.
- **Resolved specification ‚Äî Full mode (`gemini-2.5-pro`):**
  - **Field set:** everything Smart has, plus `openedAt`, `executionQuality`, `emotionalState` inline; the free-text fields (`notes`, `didRight`, `wouldChange`) are **never** sent inline regardless of trade count (a single trade can contribute up to ~12,000 chars across the three) ‚Äî always fetched via `queryTrades` when a question actually needs them.
  - **Same threshold methodology as Smart, same 60 KB rule** ‚Äî applied to Full's richer "core" field set (10 fields vs. Smart's 7), which naturally yields a lower trade-count ceiling in practice (~250-260 trades vs. ~380-400 for Smart, since each trade carries more bytes). Below threshold: send the core fields as raw JSON directly, single round trip ‚Äî no tool-use overhead for small accounts. Above threshold: same shared tool-use architecture as Smart-Pro, with Full's richer field/tool set.
  - **Baseline inline context (both above and below threshold):** the same small, fixed-size headline KPIs already computed by `computeResearchAggregates` (`stats`: winRate, avgR, expectancy, totalPnl, maxDrawdown, profitFactor, totalTrades), so trivial questions never need a tool round-trip even once Full is in its tool-use tier.
- **Shared tool-use architecture** (used by Smart-Pro above its threshold, and by Full above its threshold):
  - **`queryTrades({filters, fields, orderBy, direction, limit, offset})`** ‚Äî one general-purpose raw-row/free-text retrieval tool, replacing a longer list of bespoke drill-down tools. Default `limit` ~20 ‚Äî **not a hard ceiling the model can't exceed**, but a sane starting point; the model can request more, up to a generous per-call cap (~100-150) so a "compare all N trades" ask can't regress into the original unbounded-dump problem in disguise. The model **must** state the scope it actually used in its answer (e.g. "compared your 20 most recent vs. 20 oldest trades, out of 340 total") so the user can judge whether to ask for a wider comparison; requests beyond the per-call cap are served by pagination (`offset`) across multiple calls, not silent truncation. Fields exposed depend on mode (Smart: excludes `openedAt`/`executionQuality`/`emotionalState`/free-text; Full: all of them).
  - **Aggregation tools** ‚Äî exact server-computed summaries, justified not by "no other way to get the data" (`queryTrades` covers that) but because precise math (win rate, avg R, averages) shouldn't be left to an LLM estimating over raw rows. `getSetupBreakdown`, `getTickerBreakdown`, `getDayHourBreakdown` (`openedAt`-based, contingent on **P12**) for both modes; Full-only additions: `getExecutionQualityBreakdown`, `getEmotionalStateBreakdown`, `getHoldTimeVsRSummary`. All reuse/extend the `computeResearchAggregates` (P9) pattern. Bounded-cardinality categorical text fields (`setupType`, `emotionalState`) aggregate exactly like a number ‚Äî grouped by value ‚Äî with every `"◊ê◊ó◊® - ..."` custom value collapsed into one `◊ê◊ó◊®` bucket for aggregation purposes (the specific custom text is still reachable via `queryTrades`). Genuinely free-form text (`notes`, `didRight`, `wouldChange`) has no fixed value set and is **not** aggregatable server-side ‚Äî `queryTrades` is the only path to it.
  - **Web search tool** ‚Äî Gemini's native Search grounding, Pro-only, identical integration shared by both modes.
  - **Model self-awareness:** system prompt states what's available per mode (fields, tool categories). If a question genuinely falls outside what `queryTrades` + the aggregation tools can retrieve (data not stored at all), the model says so instead of guessing ‚Äî a rare edge case, not the default fallback, since `queryTrades` covers almost everything.
- **Shared: filter-respect toggle** ‚Äî Pro-exclusive, mode-independent (applies to both smart and full): "respect the currently-active `/research` filter in this conversation" (default: on) vs. "consider the full trade history regardless of the dashboard filter" (off). **Always visible in the UI to every user, including Free** ‚Äî Free-tier sees the toggle but it's disabled/non-interactive, locked to "respect filter" (matching today's default behavior), so the capability is discoverable rather than hidden. UI location TBD (near the mode selector in `components/chat-sidebar.tsx`).

#### [x] P2. Manual import runs sequential N+1 Supabase round-trips per leg
- **Where:** `app/api/trades/manual/route.ts:43-87`
- **Issue:** **Confirmed.** A single multi-leg submission (`/manual-import` Excel upload or batch form) runs **three sequential `for` loops**, each doing one Supabase round-trip per iteration: (1) lines 43-60 update Trade-level annotations per leg, awaited serially; (2) lines 65-80 do a `select brokerExecId` then a conditional `update source='manual'` per distinct tradeId, awaited serially; (3) lines 85-87 call `recomputeActualR` per tradeId, awaited serially. A 200-leg Excel import that touches 80 distinct trades = roughly 200 + 160 + 80 = ~440 sequential round-trips against Supabase before the route returns. Each round-trip is ~30-80ms from Vercel ‚Üí Supabase, so the response is on the order of 15-30s ‚Äî well inside the symptoms users complain about ("the import hangs"). Overlaps with [`docs/TECH-DEBT.md`](TECH-DEBT.md) ‚Äî same loop has a correctness angle (no per-leg error surfaced if one update fails).
- **Acceptance:** End-to-end POST time for a 200-leg manual import against the QA user dataset, measured server-side from request entry to response, drops to under 25% of the current sequential baseline. Verified with the Excel template in `docs/qa-test-user.md` filled to 200 legs and run 5√ó consecutively ‚Äî timings logged from `route.ts` entry / exit, before/after averages compared. Total Supabase round-trips for the route are O(distinct-trades), not O(legs) ‚Äî counted by instrumenting a request-scoped counter in `createAdminClient` for the duration of the test.

### Phase 2 ‚Äî Important (correctness & integrity)

#### [x] P4. `processExecutions` is fully sequential across executions
- **Where:** `lib/ibkr/process-executions.ts:150-158`
- **Issue:** **Confirmed.** The `for (const exec of executions)` loop awaits each `processOneExecution` in series. Each one is 1 dedup SELECT + 1 open-trade SELECT + 1-2 writes ‚Äî so a 500-execution IBKR sync (a typical month for an active user, or any backfill) is ~1500+ sequential Supabase round-trips. Same-ticker executions must stay serial ‚Äî that's the FIFO invariant in `CLAUDE.md` ("read ‚Üí match ‚Üí write cycle that retries on concurrency conflicts"), and the per-user partial unique index enforces it. But cross-ticker executions are independent: trade `AAPL` and trade `TSLA` for the same user touch disjoint rows. Grouping by ticker and processing groups in parallel (with a small concurrency cap, e.g. 5) cuts sync time by ~5√ó on typical multi-ticker datasets with no FIFO risk.
- **Acceptance:** End-to-end time for a 500-execution `processExecutions` call against the QA user, measured server-side, drops to under 40% of the current sequential baseline. Verified with a fixture of 500 executions spread across at least 20 tickers, run 3√ó to average. Per-ticker FIFO ordering is preserved ‚Äî verified by the existing FIFO + concurrency integration tests passing unchanged after the change.

#### [x] P5. IBKR cron iterates over `BrokerConnection` rows sequentially
- **Where:** `app/api/cron/ibkr-sync/route.ts:152-155`
- **Issue:** **Confirmed.** The loop is documented as sequential because "IBKR rate-limits per-token" ‚Äî but the rate limit is **per Flex token**, not global, so two different users' Flex calls are completely independent. With `maxDuration=60s` and a per-user sync taking 10-30s when IBKR is slow, ~3-4 sequential users already saturate the cron's budget; past that, later users in the list silently never get synced on this run and have to wait until the next scheduled 13:00 / 20:00 UTC tick. This becomes a real outage shape as soon as the userbase grows past the smallest handful of active connections.
- **Acceptance:** Total cron handler time for N active connections is bounded by the slowest individual sync plus a small overhead, not by the sum. Verified by mocking `syncOneConnection` to a 5s sleep and confirming a run with 8 fake connections completes in under 15s (rather than 40s). No regression in error isolation ‚Äî when one user's IBKR token fails, other users still complete successfully (verified by mixing a throwing mock among the 8).

#### [x] P6. `ChatContext` value is a fresh object literal on every provider render
- **Where:** `lib/chat/chat-context.tsx:19-32`
- **Issue:** **Confirmed.** The `ChatContext.Provider` value is an inline object literal `{ isOpen, toggleChat: () => setIsOpen(p => !p), contextData, setContextData }` and the `toggleChat` arrow is allocated fresh on every render. Any state change inside the provider invalidates the context value reference, so **every** consumer of `useChatContext()` re-renders, even consumers that only read `isOpen` and have no interest in `contextData`. Compounds P3 ‚Äî the broadcast in P3 invalidates the value, which then re-renders not just the sidebar but anything else reading the context. Two separate concerns are bundled into one context, but they have very different update frequencies (`isOpen` flips rarely, `contextData` mutates per keystroke).
- **Acceptance:** Reading `isOpen` alone does not re-render when only `contextData` changes. Verified by adding a counter component that subscribes only to `isOpen` (via a split context or an external store), typing into the research filter bar, and confirming the counter does not increment. `useMemo` / `useCallback` alone is not sufficient if the context shape stays unified ‚Äî the test only passes if the two concerns are decoupled.
